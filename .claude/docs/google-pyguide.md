This style guide is a list of <em>dos and don&#39;ts</em> for Python programs.</p><p>To help you format code correctly, we&#39;ve created a <a href="google_python_style.vim">settings file for Vim</a>. For Emacs, the default settings should be fine.</p><p>Many teams use the <a href="https://github.com/google/yapf/">yapf</a> auto-formatter to avoid arguing over formatting.</p><p> </p><p></p><h2><a class="h" name="2-Python-Language-Rules" href="#2-Python-Language-Rules"><span></span></a><a class="h" name="2-python-language-rules" href="#2-python-language-rules"><span></span></a>2 Python Language Rules</h2><p> </p><p></p><h3><a class="h" name="2_1-Lint" href="#2_1-Lint"><span></span></a><a class="h" name="2_1-lint" href="#2_1-lint"><span></span></a>2.1 Lint</h3><p>Run <code class="code">pylint</code> over your code.</p><p> </p><p></p><h4><a class="h" name="2_1_1-Definition" href="#2_1_1-Definition"><span></span></a><a class="h" name="2_1_1-definition" href="#2_1_1-definition"><span></span></a>2.1.1 Definition</h4><p><code class="code">pylint</code> is a tool for finding bugs and style problems in Python source code. It finds problems that are typically caught by a compiler for less dynamic languages like C and C++. Because of the dynamic nature of Python, some warnings may be incorrect; however, spurious warnings should be fairly infrequent.</p><p> </p><p></p><h4><a class="h" name="2_1_2-Pros" href="#2_1_2-Pros"><span></span></a><a class="h" name="2_1_2-pros" href="#2_1_2-pros"><span></span></a>2.1.2 Pros</h4><p>Catches easy-to-miss errors like typos, using-vars-before-assignment, etc.</p><p> </p><p></p><h4><a class="h" name="2_1_3-Cons" href="#2_1_3-Cons"><span></span></a><a class="h" name="2_1_3-cons" href="#2_1_3-cons"><span></span></a>2.1.3 Cons</h4><p><code class="code">pylint</code> isn&lsquo;t perfect. To take advantage of it, we&rsquo;ll need to sometimes: a) Write around it b) Suppress its warnings or c) Improve it.</p><p> </p><p></p><h4><a class="h" name="2_1_4-Decision" href="#2_1_4-Decision"><span></span></a><a class="h" name="2_1_4-decision" href="#2_1_4-decision"><span></span></a>2.1.4 Decision</h4><p>Make sure you run <code class="code">pylint</code> on your code.</p><p>Suppress warnings if they are inappropriate so that other issues are not hidden. To suppress warnings, you can set a line-level comment:</p><pre class="code"><span class="pln">dict </span><span class="pun">=</span><span class="pln"> </span><span class="str">&#39;something awful&#39;</span><span class="pln">  </span><span class="com"># Bad Idea... pylint: disable=redefined-builtin</span><span class="pln">
</span></pre><p><code class="code">pylint</code> warnings are each identified by symbolic name (<code class="code">empty-docstring</code>) Google-specific warnings start with <code class="code">g-</code>.</p><p>If the reason for the suppression is not clear from the symbolic name, add an explanation.</p><p>Suppressing in this way has the advantage that we can easily search for suppressions and revisit them.</p><p>You can get a list of <code class="code">pylint</code> warnings by doing:</p><pre class="code"><span class="pln">pylint </span><span class="pun">--</span><span class="pln">list</span><span class="pun">-</span><span class="pln">msgs
</span></pre><p>To get more information on a particular message, use:</p><pre class="code"><span class="pln">pylint </span><span class="pun">--</span><span class="pln">help</span><span class="pun">-</span><span class="pln">msg</span><span class="pun">=</span><span class="pln">C6409
</span></pre><p>Prefer <code class="code">pylint: disable</code> to the deprecated older form <code class="code">pylint: disable-msg</code>.</p><p>Unused argument warnings can be suppressed by deleting the variables at the beginning of the function. Always include a comment explaining why you are deleting it. &ldquo;Unused.&rdquo; is sufficient. For example:</p><pre class="code"><span class="kwd">def</span><span class="pln"> viking_cafe_order</span><span class="pun">(</span><span class="pln">spam</span><span class="pun">,</span><span class="pln"> beans</span><span class="pun">,</span><span class="pln"> eggs</span><span class="pun">=</span><span class="kwd">None</span><span class="pun">):</span><span class="pln">
    </span><span class="kwd">del</span><span class="pln"> beans</span><span class="pun">,</span><span class="pln"> eggs  </span><span class="com"># Unused by vikings.</span><span class="pln">
    </span><span class="kwd">return</span><span class="pln"> spam </span><span class="pun">+</span><span class="pln"> spam </span><span class="pun">+</span><span class="pln"> spam
</span></pre><p>Other common forms of suppressing this warning include using &lsquo;<code class="code">_</code>&rsquo; as the identifier for the unused argument, prefixing the argument name with &lsquo;<code class="code">unused_</code>&rsquo;, or assigning them to &lsquo;<code class="code">_</code>&rsquo;. These forms are allowed but no longer encouraged. The first two break callers that pass arguments by name, while the last does not enforce that the arguments are actually unused.</p><p> </p><p></p><h3><a class="h" name="2_2-Imports" href="#2_2-Imports"><span></span></a><a class="h" name="2_2-imports" href="#2_2-imports"><span></span></a>2.2 Imports</h3><p>Use <code class="code">import</code> statements for packages and modules only, not for individual classes or functions. Note that there is an explicit exemption for imports from the <a href="#typing-imports">typing module</a>.</p><p> </p><p></p><h4><a class="h" name="2_2_1-Definition" href="#2_2_1-Definition"><span></span></a><a class="h" name="2_2_1-definition" href="#2_2_1-definition"><span></span></a>2.2.1 Definition</h4><p>Reusability mechanism for sharing code from one module to another.</p><p> </p><p></p><h4><a class="h" name="2_2_2-Pros" href="#2_2_2-Pros"><span></span></a><a class="h" name="2_2_2-pros" href="#2_2_2-pros"><span></span></a>2.2.2 Pros</h4><p>The namespace management convention is simple. The source of each identifier is indicated in a consistent way; <code class="code">x.Obj</code> says that object <code class="code">Obj</code> is defined in module <code class="code">x</code>.</p><p> </p><p></p><h4><a class="h" name="2_2_3-Cons" href="#2_2_3-Cons"><span></span></a><a class="h" name="2_2_3-cons" href="#2_2_3-cons"><span></span></a>2.2.3 Cons</h4><p>Module names can still collide. Some module names are inconveniently long.</p><p> </p><p></p><h4><a class="h" name="2_2_4-Decision" href="#2_2_4-Decision"><span></span></a><a class="h" name="2_2_4-decision" href="#2_2_4-decision"><span></span></a>2.2.4 Decision</h4><ul><li>Use <code class="code">import x</code> for importing packages and modules.</li><li>Use <code class="code">from x import y</code> where <code class="code">x</code> is the package prefix and <code class="code">y</code> is the module name with no prefix.</li><li>Use <code class="code">from x import y as z</code> if two modules named <code class="code">y</code> are to be imported or if <code class="code">y</code> is an inconveniently long name.</li><li>Use <code class="code">import y as z</code> only when <code class="code">z</code> is a standard abbreviation (e.g., <code class="code">np</code> for <code class="code">numpy</code>).</li></ul><p>For example the module <code class="code">sound.effects.echo</code> may be imported as follows:</p><pre class="code"><span class="kwd">from</span><span class="pln"> sound</span><span class="pun">.</span><span class="pln">effects </span><span class="kwd">import</span><span class="pln"> echo
</span><span class="pun">...</span><span class="pln">
echo</span><span class="pun">.</span><span class="typ">EchoFilter</span><span class="pun">(</span><span class="pln">input</span><span class="pun">,</span><span class="pln"> output</span><span class="pun">,</span><span class="pln"> delay</span><span class="pun">=</span><span class="lit">0.7</span><span class="pun">,</span><span class="pln"> atten</span><span class="pun">=</span><span class="lit">4</span><span class="pun">)</span><span class="pln">
</span></pre><p>Do not use relative names in imports. Even if the module is in the same package, use the full package name. This helps prevent unintentionally importing a package twice.</p><p>Imports from the <a href="#typing-imports">typing module</a> and the <a href="https://six.readthedocs.io/#module-six.moves">six.moves module</a> are exempt from this rule.</p><p> </p><p></p><h3><a class="h" name="2_3-Packages" href="#2_3-Packages"><span></span></a><a class="h" name="2_3-packages" href="#2_3-packages"><span></span></a>2.3 Packages</h3><p>Import each module using the full pathname location of the module.</p><p> </p><p></p><h4><a class="h" name="2_3_1-Pros" href="#2_3_1-Pros"><span></span></a><a class="h" name="2_3_1-pros" href="#2_3_1-pros"><span></span></a>2.3.1 Pros</h4><p>Avoids conflicts in module names or incorrect imports due to the module search path not being what the author expected.  Makes it easier to find modules.</p><p> </p><p></p><h4><a class="h" name="2_3_2-Cons" href="#2_3_2-Cons"><span></span></a><a class="h" name="2_3_2-cons" href="#2_3_2-cons"><span></span></a>2.3.2 Cons</h4><p>Makes it harder to deploy code because you have to replicate the package hierarchy.  Not really a problem with modern deployment mechanisms.</p><p> </p><p></p><h4><a class="h" name="2_3_3-Decision" href="#2_3_3-Decision"><span></span></a><a class="h" name="2_3_3-decision" href="#2_3_3-decision"><span></span></a>2.3.3 Decision</h4><p>All new code should import each module by its full package name.</p><p>Imports should be as follows:</p><p>Yes:</p><pre class="code"><span class="com"># Reference absl.flags in code with the complete name (verbose).</span><span class="pln">
</span><span class="kwd">import</span><span class="pln"> absl</span><span class="pun">.</span><span class="pln">flags
</span><span class="kwd">from</span><span class="pln"> doctor</span><span class="pun">.</span><span class="pln">who </span><span class="kwd">import</span><span class="pln"> jodie

FLAGS </span><span class="pun">=</span><span class="pln"> absl</span><span class="pun">.</span><span class="pln">flags</span><span class="pun">.</span><span class="pln">FLAGS
</span></pre><pre class="code"><span class="com"># Reference flags in code with just the module name (common).</span><span class="pln">
</span><span class="kwd">from</span><span class="pln"> absl </span><span class="kwd">import</span><span class="pln"> flags
</span><span class="kwd">from</span><span class="pln"> doctor</span><span class="pun">.</span><span class="pln">who </span><span class="kwd">import</span><span class="pln"> jodie

FLAGS </span><span class="pun">=</span><span class="pln"> flags</span><span class="pun">.</span><span class="pln">FLAGS
</span></pre><p>No: <em>(assume this file lives in <code class="code">doctor/who/</code> where <code class="code">jodie.py</code> also exists)</em></p><pre class="code"><span class="com"># Unclear what module the author wanted and what will be imported.  The actual</span><span class="pln">
</span><span class="com"># import behavior depends on external factors controlling sys.path.</span><span class="pln">
</span><span class="com"># Which possible jodie module did the author intend to import?</span><span class="pln">
</span><span class="kwd">import</span><span class="pln"> jodie
</span></pre><p>The directory the main binary is located in should not be assumed to be in <code class="code">sys.path</code> despite that happening in some environments.  This being the case, code should assume that <code class="code">import jodie</code> refers to a third party or top level package named <code class="code">jodie</code>, not a local <code class="code">jodie.py</code>.</p><p> </p><p></p><h3><a class="h" name="2_4-Exceptions" href="#2_4-Exceptions"><span></span></a><a class="h" name="2_4-exceptions" href="#2_4-exceptions"><span></span></a>2.4 Exceptions</h3><p>Exceptions are allowed but must be used carefully.</p><p> </p><p></p><h4><a class="h" name="2_4_1-Definition" href="#2_4_1-Definition"><span></span></a><a class="h" name="2_4_1-definition" href="#2_4_1-definition"><span></span></a>2.4.1 Definition</h4><p>Exceptions are a means of breaking out of the normal flow of control of a code block to handle errors or other exceptional conditions.</p><p> </p><p></p><h4><a class="h" name="2_4_2-Pros" href="#2_4_2-Pros"><span></span></a><a class="h" name="2_4_2-pros" href="#2_4_2-pros"><span></span></a>2.4.2 Pros</h4><p>The control flow of normal operation code is not cluttered by error-handling code. It also allows the control flow to skip multiple frames when a certain condition occurs, e.g., returning from N nested functions in one step instead of having to carry-through error codes.</p><p> </p><p></p><h4><a class="h" name="2_4_3-Cons" href="#2_4_3-Cons"><span></span></a><a class="h" name="2_4_3-cons" href="#2_4_3-cons"><span></span></a>2.4.3 Cons</h4><p>May cause the control flow to be confusing. Easy to miss error cases when making library calls.</p><p> </p><p></p><h4><a class="h" name="2_4_4-Decision" href="#2_4_4-Decision"><span></span></a><a class="h" name="2_4_4-decision" href="#2_4_4-decision"><span></span></a>2.4.4 Decision</h4><p>Exceptions must follow certain conditions:</p><ul><li><p>Raise exceptions like this: <code class="code">raise MyError(&#39;Error message&#39;)</code> or <code class="code">raise MyError()</code>. Do not use the two-argument form (<code class="code">raise MyError, &#39;Error message&#39;</code>).</p></li><li><p>Make use of built-in exception classes when it makes sense. For example, raise a <code class="code">ValueError</code> to indicate a programming mistake like a violated precondition (such as if you were passed a negative number but required a positive one). Do not use <code class="code">assert</code> statements for validating argument values of a public API. <code class="code">assert</code> is used to ensure internal correctness, not to enforce correct usage nor to indicate that some unexpected event occurred. If an exception is desired in the latter cases, use a raise statement. For example:</p><pre class="code"><span class="typ">Yes</span><span class="pun">:</span><span class="pln">
  </span><span class="kwd">def</span><span class="pln"> connect_to_next_port</span><span class="pun">(</span><span class="pln">self</span><span class="pun">,</span><span class="pln"> minimum</span><span class="pun">):</span><span class="pln">
    </span><span class="str">&quot;&quot;&quot;Connects to the next available port.

    Args:
      minimum: A port value greater or equal to 1024.

    Returns:
      The new minimum port.

    Raises:
      ConnectionError: If no available port is found.
    &quot;&quot;&quot;</span><span class="pln">
    </span><span class="kwd">if</span><span class="pln"> minimum </span><span class="pun">&lt;</span><span class="pln"> </span><span class="lit">1024</span><span class="pun">:</span><span class="pln">
      </span><span class="com"># Note that this raising of ValueError is not mentioned in the doc</span><span class="pln">
      </span><span class="com"># string&#39;s &quot;Raises:&quot; section because it is not appropriate to</span><span class="pln">
      </span><span class="com"># guarantee this specific behavioral reaction to API misuse.</span><span class="pln">
      </span><span class="kwd">raise</span><span class="pln"> </span><span class="typ">ValueError</span><span class="pun">(</span><span class="str">&#39;Minimum port must be at least 1024, not %d.&#39;</span><span class="pln"> </span><span class="pun">%</span><span class="pln"> </span><span class="pun">(</span><span class="pln">minimum</span><span class="pun">,))</span><span class="pln">
    port </span><span class="pun">=</span><span class="pln"> self</span><span class="pun">.</span><span class="pln">_find_next_open_port</span><span class="pun">(</span><span class="pln">minimum</span><span class="pun">)</span><span class="pln">
    </span><span class="kwd">if</span><span class="pln"> </span><span class="kwd">not</span><span class="pln"> port</span><span class="pun">:</span><span class="pln">
      </span><span class="kwd">raise</span><span class="pln"> </span><span class="typ">ConnectionError</span><span class="pun">(</span><span class="str">&#39;Could not connect to service on %d or higher.&#39;</span><span class="pln"> </span><span class="pun">%</span><span class="pln"> </span><span class="pun">(</span><span class="pln">minimum</span><span class="pun">,))</span><span class="pln">
    </span><span class="kwd">assert</span><span class="pln"> port </span><span class="pun">&gt;=</span><span class="pln"> minimum</span><span class="pun">,</span><span class="pln"> </span><span class="str">&#39;Unexpected port %d when minimum was %d.&#39;</span><span class="pln"> </span><span class="pun">%</span><span class="pln"> </span><span class="pun">(</span><span class="pln">port</span><span class="pun">,</span><span class="pln"> minimum</span><span class="pun">)</span><span class="pln">
    </span><span class="kwd">return</span><span class="pln"> port
</span></pre><pre class="code"><span class="typ">No</span><span class="pun">:</span><span class="pln">
  </span><span class="kwd">def</span><span class="pln"> connect_to_next_port</span><span class="pun">(</span><span class="pln">self</span><span class="pun">,</span><span class="pln"> minimum</span><span class="pun">):</span><span class="pln">
    </span><span class="str">&quot;&quot;&quot;Connects to the next available port.

    Args:
      minimum: A port value greater or equal to 1024.

    Returns:
      The new minimum port.
    &quot;&quot;&quot;</span><span class="pln">
    </span><span class="kwd">assert</span><span class="pln"> minimum </span><span class="pun">&gt;=</span><span class="pln"> </span><span class="lit">1024</span><span class="pun">,</span><span class="pln"> </span><span class="str">&#39;Minimum port must be at least 1024.&#39;</span><span class="pln">
    port </span><span class="pun">=</span><span class="pln"> self</span><span class="pun">.</span><span class="pln">_find_next_open_port</span><span class="pun">(</span><span class="pln">minimum</span><span class="pun">)</span><span class="pln">
    </span><span class="kwd">assert</span><span class="pln"> port </span><span class="kwd">is</span><span class="pln"> </span><span class="kwd">not</span><span class="pln"> </span><span class="kwd">None</span><span class="pln">
    </span><span class="kwd">return</span><span class="pln"> port
</span></pre></li><li><p>Libraries or packages may define their own exceptions. When doing so they must inherit from an existing exception class. Exception names should end in <code class="code">Error</code> and should not introduce stutter (<code class="code">foo.FooError</code>).</p></li><li><p>Never use catch-all <code class="code">except:</code> statements, or catch <code class="code">Exception</code> or <code class="code">StandardError</code>, unless you are</p><ul><li>re-raising the exception, or</li><li>creating an isolation point in the program where exceptions are not propagated but are recorded and suppressed instead, such as protecting a thread from crashing by guarding its outermost block.</li></ul><p>Python is very tolerant in this regard and <code class="code">except:</code> will really catch everything including misspelled names, sys.exit() calls, Ctrl+C interrupts, unittest failures and all kinds of other exceptions that you simply don&#39;t want to catch.</p></li><li><p>Minimize the amount of code in a <code class="code">try</code>/<code class="code">except</code> block. The larger the body of the <code class="code">try</code>, the more likely that an exception will be raised by a line of code that you didn&#39;t expect to raise an exception. In those cases, the <code class="code">try</code>/<code class="code">except</code> block hides a real error.</p></li><li><p>Use the <code class="code">finally</code> clause to execute code whether or not an exception is raised in the <code class="code">try</code> block. This is often useful for cleanup, i.e., closing a file.</p></li><li><p>When capturing an exception, use <code class="code">as</code> rather than a comma. For example:</p><pre class="code"><span class="kwd">try</span><span class="pun">:</span><span class="pln">
  </span><span class="kwd">raise</span><span class="pln"> </span><span class="typ">Error</span><span class="pun">()</span><span class="pln">
</span><span class="kwd">except</span><span class="pln"> </span><span class="typ">Error</span><span class="pln"> </span><span class="kwd">as</span><span class="pln"> error</span><span class="pun">:</span><span class="pln">
  </span><span class="kwd">pass</span><span class="pln">
</span></pre></li></ul><p> </p><p></p><h3><a class="h" name="2_5-Global-variables" href="#2_5-Global-variables"><span></span></a><a class="h" name="2_5-global-variables" href="#2_5-global-variables"><span></span></a>2.5 Global variables</h3><p>Avoid global variables.</p><p> </p><p></p><h4><a class="h" name="2_5_1-Definition" href="#2_5_1-Definition"><span></span></a><a class="h" name="2_5_1-definition" href="#2_5_1-definition"><span></span></a>2.5.1 Definition</h4><p>Variables that are declared at the module level or as class attributes.</p><p> </p><p></p><h4><a class="h" name="2_5_2-Pros" href="#2_5_2-Pros"><span></span></a><a class="h" name="2_5_2-pros" href="#2_5_2-pros"><span></span></a>2.5.2 Pros</h4><p>Occasionally useful.</p><p> </p><p></p><h4><a class="h" name="2_5_3-Cons" href="#2_5_3-Cons"><span></span></a><a class="h" name="2_5_3-cons" href="#2_5_3-cons"><span></span></a>2.5.3 Cons</h4><p>Has the potential to change module behavior during the import, because assignments to global variables are done when the module is first imported.</p><p> </p><p></p><h4><a class="h" name="2_5_4-Decision" href="#2_5_4-Decision"><span></span></a><a class="h" name="2_5_4-decision" href="#2_5_4-decision"><span></span></a>2.5.4 Decision</h4><p>Avoid global variables.</p><p>While they are technically variables, module-level constants are permitted and encouraged. For example: <code class="code">MAX_HOLY_HANDGRENADE_COUNT = 3</code>. Constants must be named using all caps with underscores. See <a href="#s3.16-naming">Naming</a> below.</p><p>If needed, globals should be declared at the module level and made internal to the module by prepending an <code class="code">_</code> to the name. External access must be done through public module-level functions. See <a href="#s3.16-naming">Naming</a> below.</p><p> </p><p></p><h3><a class="h" name="2_6-Nested_Local_Inner-Classes-and-Functions" href="#2_6-Nested_Local_Inner-Classes-and-Functions"><span></span></a><a class="h" name="2_6-nested_local_inner-classes-and-functions" href="#2_6-nested_local_inner-classes-and-functions"><span></span></a>2.6 Nested/Local/Inner Classes and Functions</h3><p>Nested local functions or classes are fine when used to close over a local variable. Inner classes are fine.</p><p> </p><p></p><h4><a class="h" name="2_6_1-Definition" href="#2_6_1-Definition"><span></span></a><a class="h" name="2_6_1-definition" href="#2_6_1-definition"><span></span></a>2.6.1 Definition</h4><p>A class can be defined inside of a method, function, or class. A function can be defined inside a method or function. Nested functions have read-only access to variables defined in enclosing scopes.</p><p> </p><p></p><h4><a class="h" name="2_6_2-Pros" href="#2_6_2-Pros"><span></span></a><a class="h" name="2_6_2-pros" href="#2_6_2-pros"><span></span></a>2.6.2 Pros</h4><p>Allows definition of utility classes and functions that are only used inside of a very limited scope. Very <a href="http://www.google.com/url?sa=D&amp;q=http://en.wikipedia.org/wiki/Abstract_data_type">ADT</a>-y. Commonly used for implementing decorators.</p><p> </p><p></p><h4><a class="h" name="2_6_3-Cons" href="#2_6_3-Cons"><span></span></a><a class="h" name="2_6_3-cons" href="#2_6_3-cons"><span></span></a>2.6.3 Cons</h4><p>Instances of nested or local classes cannot be pickled. Nested functions and classes cannot be directly tested. Nesting can make your outer function longer and less readable.</p><p> </p><p></p><h4><a class="h" name="2_6_4-Decision" href="#2_6_4-Decision"><span></span></a><a class="h" name="2_6_4-decision" href="#2_6_4-decision"><span></span></a>2.6.4 Decision</h4><p>They are fine with some caveats. Avoid nested functions or classes except when closing over a local value. Do not nest a function just to hide it from users of a module. Instead, prefix its name with an _ at the module level so that it can still be accessed by tests.</p><p>   </p><p></p><h3><a class="h" name="2_7-Comprehensions-Generator-Expressions" href="#2_7-Comprehensions-Generator-Expressions"><span></span></a><a class="h" name="2_7-comprehensions-generator-expressions" href="#2_7-comprehensions-generator-expressions"><span></span></a>2.7 Comprehensions &amp; Generator Expressions</h3><p>Okay to use for simple cases.</p><p> </p><p></p><h4><a class="h" name="2_7_1-Definition" href="#2_7_1-Definition"><span></span></a><a class="h" name="2_7_1-definition" href="#2_7_1-definition"><span></span></a>2.7.1 Definition</h4><p>List, Dict, and Set comprehensions as well as generator expressions provide a concise and efficient way to create container types and iterators without resorting to the use of traditional loops, <code class="code">map()</code>, <code class="code">filter()</code>, or <code class="code">lambda</code>.</p><p> </p><p></p><h4><a class="h" name="2_7_2-Pros" href="#2_7_2-Pros"><span></span></a><a class="h" name="2_7_2-pros" href="#2_7_2-pros"><span></span></a>2.7.2 Pros</h4><p>Simple comprehensions can be clearer and simpler than other dict, list, or set creation techniques. They are often used to define callbacks or operators for higher-order functions like <code class="code">map()</code> and <code class="code">filter()</code>.</p><p> </p><p></p><h4><a class="h" name="2_10_2-Pros" href="#2_10_2-Pros"><span></span></a><a class="h" name="2_10_2-pros" href="#2_10_2-pros"><span></span></a>2.10.2 Pros</h4><p>Convenient.</p><p> </p><p></p><h4><a class="h" name="2_10_3-Cons" href="#2_10_3-Cons"><span></span></a><a class="h" name="2_10_3-cons" href="#2_10_3-cons"><span></span></a>2.10.3 Cons</h4><p>Harder to read and debug than local functions.      map</span><span class="pun">(</span><span class="pln">math</span><span class="pun">.</span><span class="pln">sqrt</span><span class="pun">,</span><span class="pln"> data</span><span class="pun">)</span><span class="pln">    </span><span class="com"># Ok. No inlined lambda expression.</span><span class="pln">

     fn</span><span class="pun">(*</span><span class="pln">args</span><span class="pun">,</span><span class="pln"> </span><span class="pun">**</span><span class="pln">kwargs</span><span class="pun">)</span><span class="pln">
</span></pre><pre class="code"><span class="typ">No</span><span class="pun">:</span><span class="pln">  words </span><span class="pun">=</span><span class="pln"> string</span><span class="pun">.</span><span class="pln">split</span><span class="pun">(</span><span class="pln">foo</span><span class="pun">,</span><span class="pln"> </span><span class="str">&#39;:&#39;</span><span class="pun">)</span><span class="pln">

     map</span><span class="pun">(</span><span class="kwd">lambda</span><span class="pln"> x</span><span class="pun">:</span><span class="pln"> x</span><span class="pun">[</span><span class="lit">1</span><span class="pun">],</span><span class="pln"> filter</span><span class="pun">(</span><span class="kwd">lambda</span><span class="pln"> x</span><span class="pun">:</span><span class="pln"> x</span><span class="pun">[</span><span class="lit">2</span><span class="pun">]</span><span class="pln"> </span><span class="pun">==</span><span class="pln"> </span><span class="lit">5</span><span class="pun">,</span><span class="pln"> my_list</span><span class="pun">))</span><span class="pln">

     apply</span><span class="pun">(</span><span class="pln">fn</span><span class="pun">,</span><span class="pln"> args</span><span class="pun">,</span><span class="pln"> kwargs</span><span class="pun">)</span><span class="pln">
</span></pre><p> </p><p></p><h3><a class="h" name="2_16-Lexical-Scoping" href="#2_16-Lexical-Scoping"><span></span></a><a class="h" name="2_16-lexical-scoping" href="#2_16-lexical-scoping"><span></span></a>2.16 Lexical Scoping</h3><p>Okay to use.</p><p> </p><p></p><h4><a class="h" name="2_16_1-Definition" href="#2_16_1-Definition"><span></span></a><a class="h" name="2_16_1-definition" href="#2_16_1-definition"><span></span></a>2.16.1 Definition</h4><p>A nested Python function can refer to variables defined in enclosing functions, but can not assign to them. Variable bindings are resolved using lexical scoping, that is, based on the static program text. Any assignment to a name in a block will cause Python to treat all references to that name as a local variable, even if the use precedes the assignment. If a global declaration occurs, the name is treated as a global variable.</p><p>An example of the use of this feature is:</p><pre class="code"><span class="kwd">def</span><span class="pln"> get_adder</span><span class="pun">(</span><span class="pln">summand1</span><span class="pun">):</span><span class="pln">
    </span><span class="str">&quot;&quot;&quot;Returns a function that adds numbers to a given number.&quot;&quot;&quot;</span><span class="pln">
    </span><span class="kwd">def</span><span class="pln"> adder</span><span class="pun">(</span><span class="pln">summand2</span><span class="pun">):</span><span class="pln">
        </span><span class="kwd">return</span><span class="pln"> summand1 </span><span class="pun">+</span><span class="pln"> summand2

    </span><span class="kwd">return</span><span class="pln"> adder
</span></pre><p> </p><p></p><h4><a class="h" name="2_16_2-Pros" href="#2_16_2-Pros"><span></span></a><a class="h" name="2_16_2-pros" href="#2_16_2-pros"><span></span></a>2.16.2 Pros</h4><p>Often results in clearer, more elegant code. Especially comforting to experienced Lisp and Scheme (and Haskell and ML and ...) programmers.</p><p> </p><p></p><h4><a class="h" name="2_16_3-Cons" href="#2_16_3-Cons"><span></span></a><a class="h" name="2_16_3-cons" href="#2_16_3-cons"><span></span></a>2.16.3 Cons</h4><p>Can lead to confusing bugs. Such as this example based on <a href="http://www.google.com/url?sa=D&amp;q=http://www.python.org/dev/peps/pep-0227/">PEP-0227</a>:</p><pre class="code"><span class="pln">i </span><span class="pun">=</span><span class="pln"> </span><span class="lit">4</span><span class="pln">
</span><span class="kwd">def</span><span class="pln"> foo</span><span class="pun">(</span><span class="pln">x</span><span class="pun">):</span><span class="pln">
    </span><span class="kwd">def</span><span class="pln"> bar</span><span class="pun">():</span><span class="pln">
        </span><span class="kwd">print</span><span class="pun">(</span><span class="pln">i</span><span class="pun">,</span><span class="pln"> end</span><span class="pun">=</span><span class="str">&#39;&#39;</span><span class="pun">)</span><span class="pln">
    </span><span class="com"># ...</span><span class="pln">
    </span><span class="com"># A bunch of code here</span><span class="pln">
    </span><span class="com"># ...</span><span class="pln">
    </span><span class="kwd">for</span><span class="pln"> i </span><span class="kwd">in</span><span class="pln"> x</span><span class="pun">:</span><span class="pln">  </span><span class="com"># Ah, i *is* local to foo, so this is what bar sees</span><span class="pln">
        </span><span class="kwd">print</span><span class="pun">(</span><span class="pln">i</span><span class="pun">,</span><span class="pln"> end</span><span class="pun">=</span><span class="str">&#39;&#39;</span><span class="pun">)</span><span class="pln">
    bar</span><span class="pun">()</span><span class="pln">
</span></pre><p>So <code class="code">foo([1, 2, 3])</code> will print <code class="code">1 2 3 3</code>, not <code class="code">1 2 3 4</code>.</p><p> </p><p></p><h4><a class="h" name="2_16_4-Decision" href="#2_16_4-Decision"><span></span></a><a class="h" name="2_16_4-decision" href="#2_16_4-decision"><span></span></a>2.16.4 Decision</h4><p>Okay to use.</p><p>  </p><p></p><h3><a class="h" name="2_17-Function-and-Method-Decorators" href="#2_17-Function-and-Method-Decorators"><span></span></a><a class="h" name="2_17-function-and-method-decorators" href="#2_17-function-and-method-decorators"><span></span></a>2.17 Function and Method Decorators</h3><p>Use decorators judiciously when there is a clear advantage. Avoid <code class="code">@staticmethod</code> and limit use of <code class="code">@classmethod</code>.</p><p> </p><p></p><h4><a class="h" name="2_17_1-Definition" href="#2_17_1-Definition"><span></span></a><a class="h" name="2_17_1-definition" href="#2_17_1-definition"><span></span></a>2.17.1 Definition</h4><p><a href="https://docs.python.org/3/glossary.html#term-decorator">Decorators for Functions and Methods</a> (a.k.a &ldquo;the <code class="code">@</code> notation&rdquo;). One common decorator is <code class="code">@property</code>, used for converting ordinary methods into dynamically computed attributes. However, the decorator syntax allows for user-defined decorators as well. Specifically, for some function <code class="code">my_decorator</code>, this:</p><pre class="code"><span class="kwd">class</span><span class="pln"> C</span><span class="pun">(</span><span class="pln">object</span><span class="pun">):</span><span class="pln">
    </span><span class="lit">@my_decorator</span><span class="pln">
    </span><span class="kwd">def</span><span class="pln"> method</span><span class="pun">(</span><span class="pln">self</span><span class="pun">):</span><span class="pln">
        </span><span class="com"># method body ...</span><span class="pln">
</span></pre><p>is equivalent to:</p><pre class="code"><span class="kwd">class</span><span class="pln"> C</span><span class="pun">(</span><span class="pln">object</span><span class="pun">):</span><span class="pln">
    </span><span class="kwd">def</span><span class="pln"> method</span><span class="pun">(</span><span class="pln">self</span><span class="pun">):</span><span class="pln">
        </span><span class="com"># method body ...</span><span class="pln">
    method </span><span class="pun">=</span><span class="pln"> my_decorator</span><span class="pun">(</span><span class="pln">method</span><span class="pun">)</span><span class="pln">
</span></pre><p> </p><p></p><h4><a class="h" name="2_17_2-Pros" href="#2_17_2-Pros"><span></span></a><a class="h" name="2_17_2-pros" href="#2_17_2-pros"><span></span></a>2.17.2 Pros</h4><p>Elegantly specifies some transformation on a method; the transformation might eliminate some repetitive code, enforce invariants, etc.</p><p> </p><p></p><h4><a class="h" name="2_17_3-Cons" href="#2_17_3-Cons"><span></span></a><a class="h" name="2_17_3-cons" href="#2_17_3-cons"><span></span></a>2.17.3 Cons</h4><p>Decorators can perform arbitrary operations on a function&#39;s arguments or return values, resulting in surprising implicit behavior. Additionally, decorators execute at import time. Failures in decorator code are pretty much impossible to recover from.</p><p> </p><p></p><h4><a class="h" name="2_17_4-Decision" href="#2_17_4-Decision"><span></span></a><a class="h" name="2_17_4-decision" href="#2_17_4-decision"><span></span></a>2.17.4 Decision</h4><p>Use decorators judiciously when there is a clear advantage. Decorators should follow the same import and naming guidelines as functions. Decorator pydoc should clearly state that the function is a decorator. Write unit tests for decorators.</p><p>Avoid external dependencies in the decorator itself (e.g. don&#39;t rely on files, sockets, database connections, etc.), since they might not be available when the decorator runs (at import time, perhaps from <code class="code">pydoc</code> or other tools). A decorator that is called with valid parameters should (as much as possible) be guaranteed to succeed in all cases.</p><p>Decorators are a special case of &ldquo;top level code&rdquo; - see <a href="#s3.17-main">main</a> for more discussion.</p><p>Never use <code class="code">@staticmethod</code> unless forced to in order to integrate with an API defined in an existing library. Write a module level function instead.</p><p>Use <code class="code">@classmethod</code> only when writing a named constructor or a class-specific routine that modifies necessary global state such as a process-wide cache.</p><p> </p><p></p><h3><a class="h" name="2_18-Threading" href="#2_18-Threading"><span></span></a><a class="h" name="2_18-threading" href="#2_18-threading"><span></span></a>2.18 Threading</h3><p>Do not rely on the atomicity of built-in types.</p><p>While Python&lsquo;s built-in data types such as dictionaries appear to have atomic operations, there are corner cases where they aren&rsquo;t atomic (e.g. if <code class="code">__hash__</code> or <code class="code">__eq__</code> are implemented as Python methods) and their atomicity should not be relied upon. Neither should you rely on atomic variable assignment (since this in turn depends on dictionaries).</p><p>Use the Queue module&#39;s <code class="code">Queue</code> data type as the preferred way to communicate data between threads. Otherwise, use the threading module and its locking primitives. Learn about the proper use of condition variables so you can use <code class="code">threading.Condition</code> instead of using lower-level locks.</p><p> </p><p></p><h3><a class="h" name="2_19-Power-Features" href="#2_19-Power-Features"><span></span></a><a class="h" name="2_19-power-features" href="#2_19-power-features"><span></span></a>2.19 Power Features</h3><p>Avoid these features.</p><p> </p><p></p><h4><a class="h" name="2_19_1-Definition" href="#2_19_1-Definition"><span></span></a><a class="h" name="2_19_1-definition" href="#2_19_1-definition"><span></span></a>2.19.1 Definition</h4><p>Python is an extremely flexible language and gives you many fancy features such as custom metaclasses, access to bytecode, on-the-fly compilation, dynamic inheritance, object reparenting, import hacks, reflection (e.g. some uses of <code class="code">getattr()</code>), modification of system internals, etc.</p><p> </p><p></p><h4><a class="h" name="2_19_2-Pros" href="#2_19_2-Pros"><span></span></a><a class="h" name="2_19_2-pros" href="#2_19_2-pros"><span></span></a>2.19.2 Pros</h4><p>These are powerful language features. They can make your code more compact.</p><p> </p><p></p><h4><a class="h" name="2_19_3-Cons" href="#2_19_3-Cons"><span></span></a><a class="h" name="2_19_3-cons" href="#2_19_3-cons"><span></span></a>2.19.3 Cons</h4><p>It&lsquo;s very tempting to use these &ldquo;cool&rdquo; features when they&rsquo;re not absolutely necessary. It&lsquo;s harder to read, understand, and debug code that&rsquo;s using unusual features underneath. It doesn&#39;t seem that way at first (to the original author), but when revisiting the code, it tends to be more difficult than code that is longer but is straightforward.</p><p> </p><p></p><h4><a class="h" name="2_19_4-Decision" href="#2_19_4-Decision"><span></span></a><a class="h" name="2_19_4-decision" href="#2_19_4-decision"><span></span></a>2.19.4 Decision</h4><p>Avoid these features in your code.</p><p>Standard library modules and classes that internally use these features are okay to use (for example, <code class="code">abc.ABCMeta</code>, <code class="code">collections.namedtuple</code>, <code class="code">dataclasses</code>, and <code class="code">enum</code>).</p><p> </p><p></p><h3><a class="h" name="2_20-Modern-Python_Python-3-and-from-future_imports" href="#2_20-Modern-Python_Python-3-and-from-future_imports"><span></span></a><a class="h" name="2_20-modern-python_python-3-and-from-future_imports" href="#2_20-modern-python_python-3-and-from-future_imports"><span></span></a>2.20 Modern Python: Python 3 and from __future__ imports</h3><p>Python 3 is here! While not every project is ready to use it yet, all code should be written to be 3 compatible (and tested under 3 when possible).</p><p> </p><p></p><h4><a class="h" name="2_20_1-Definition" href="#2_20_1-Definition"><span></span></a><a class="h" name="2_20_1-definition" href="#2_20_1-definition"><span></span></a>2.20.1 Definition</h4><p>Python 3 is a significant change in the Python language. While existing code is often written with 2.7 in mind, there are some simple things to do to make code more explicit about its intentions and thus better prepared for use under Python 3 without modification.</p><p> </p><p></p><h4><a class="h" name="2_20_2-Pros" href="#2_20_2-Pros"><span></span></a><a class="h" name="2_20_2-pros" href="#2_20_2-pros"><span></span></a>2.20.2 Pros</h4><p>Code written with Python 3 in mind is more explicit and easier to get running under Python 3 once all of the dependencies of your project are ready.</p><p> </p><p></p><h4><a class="h" name="2_20_3-Cons" href="#2_20_3-Cons"><span></span></a><a class="h" name="2_20_3-cons" href="#2_20_3-cons"><span></span></a>2.20.3 Cons</h4><p>Some people find the additional boilerplate to be ugly. It&lsquo;s unusual to add imports to a module that doesn&rsquo;t actually require the features added by the import.</p><p> </p><p></p><h4><a class="h" name="2_20_4-Decision" href="#2_20_4-Decision"><span></span></a><a class="h" name="2_20_4-decision" href="#2_20_4-decision"><span></span></a>2.20.4 Decision</h4><h5><a class="h" name="from-future_imports" href="#from-future_imports"><span></span></a>from __future__ imports</h5><p>Use of <code class="code">from __future__ import</code> statements is encouraged. All new code should contain the following and existing code should be updated to be compatible when possible:</p><pre class="code"><span class="kwd">from</span><span class="pln"> __future__ </span><span class="kwd">import</span><span class="pln"> absolute_import
</span><span class="kwd">from</span><span class="pln"> __future__ </span><span class="kwd">import</span><span class="pln"> division
</span><span class="kwd">from</span><span class="pln"> __future__ </span><span class="kwd">import</span><span class="pln"> print_function
</span></pre><p>If you are not already familiar with those, read up on each here: <a href="https://www.python.org/dev/peps/pep-0328/">absolute imports</a>, <a href="https://www.python.org/dev/peps/pep-0238/">new <code class="code">/</code> division behavior</a>, and <a href="https://www.python.org/dev/peps/pep-3105/">the print function</a>.</p><p>Please don&lsquo;t omit or remove these imports, even if they&rsquo;re not currently used in the module, unless the code is Python 3 only. It is better to always have the future imports in all files so that they are not forgotten during later edits when someone starts using such a feature.</p><p>There are other <code class="code">from __future__</code> import statements. Use them as you see fit. We do not include <code class="code">unicode_literals</code> in our recommendations as it is not a clear win due to implicit default codec conversion consequences it introduces in many places within Python 2.7. Most code is better off with explicit use of <code class="code">b&#39;&#39;</code> and <code class="code">u&#39;&#39;</code> bytes and unicode string literals as necessary.</p><h5><a class="h" name="The-six_future_or-past-libraries" href="#The-six_future_or-past-libraries"><span></span></a><a class="h" name="the-six_future_or-past-libraries" href="#the-six_future_or-past-libraries"><span></span></a>The six, future, or past libraries</h5><p>When your project needs to actively support use under both Python 2 and 3, use the <a href="https://pypi.org/project/six/">six</a>, <a href="https://pypi.org/project/future/">future</a>, and <a href="https://pypi.org/project/past/">past</a> libraries as you see fit. They exist to make your code cleaner and life easier.</p><p><a name="s2.21-typed-code"></a>  </p><p></p><h3><a class="h" name="2_21-Type-Annotated-Code" href="#2_21-Type-Annotated-Code"><span></span></a><a class="h" name="2_21-type-annotated-code" href="#2_21-type-annotated-code"><span></span></a>2.21 Type Annotated Code</h3><p>You can annotate Python 3 code with type hints according to <a href="https://www.python.org/dev/peps/pep-0484/">PEP-484</a>, and type-check the code at build time with a type checking tool like <a href="https://github.com/google/pytype">pytype</a>.</p><p>Type annotations can be in the source or in a <a href="https://www.python.org/dev/peps/pep-0484/#stub-files">stub pyi file</a>. Whenever possible, annotations should be in the source. Use pyi files for third-party or extension modules.</p><p> </p><p></p><h4><a class="h" name="2_21_1-Definition" href="#2_21_1-Definition"><span></span></a><a class="h" name="2_21_1-definition" href="#2_21_1-definition"><span></span></a>2.21.1 Definition</h4><p>Type annotations (or &ldquo;type hints&rdquo;) are for function or method arguments and return values:</p><pre class="code"><span class="kwd">def</span><span class="pln"> func</span><span class="pun">(</span><span class="pln">a</span><span class="pun">:</span><span class="pln"> int</span><span class="pun">)</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> </span><span class="typ">List</span><span class="pun">[</span><span class="pln">int</span><span class="pun">]:</span><span class="pln">
</span></pre><p>You can also declare the type of a variable using a special comment:</p><pre class="code"><span class="pln">a </span><span class="pun">=</span><span class="pln"> </span><span class="typ">SomeFunc</span><span class="pun">()</span><span class="pln">  </span><span class="com"># type: SomeType</span><span class="pln">
</span></pre><p> </p><p></p><h4><a class="h" name="2_21_2-Pros" href="#2_21_2-Pros"><span></span></a><a class="h" name="2_21_2-pros" href="#2_21_2-pros"><span></span></a>2.21.2 Pros</h4><p>Type annotations improve the readability and maintainability of your code. The type checker will convert many runtime errors to build-time errors, and reduce your ability to use <a href="#power-features">Power Features</a>.</p><p> </p><p></p><h4><a class="h" name="2_21_3-Cons" href="#2_21_3-Cons"><span></span></a><a class="h" name="2_21_3-cons" href="#2_21_3-cons"><span></span></a>2.21.3 Cons</h4><p>You will have to keep the type declarations up to date. You might see type errors that you think are valid code. Use of a <a href="https://github.com/google/pytype">type checker</a> may reduce your ability to use <a href="#power-features">Power Features</a>.</p><p> </p><p></p><h4><a class="h" name="2_21_4-Decision" href="#2_21_4-Decision"><span></span></a><a class="h" name="2_21_4-decision" href="#2_21_4-decision"><span></span></a>2.21.4 Decision</h4><p>You are strongly encouraged to enable Python type analysis when updating code. When adding or modifying public APIs, include type annotations and enable checking via pytype in the build system. As static analysis is relatively new to Python, we acknowledge that undesired side-effects (such as wrongly inferred types) may prevent adoption by some projects. In those situations, authors are encouraged to add a comment with a TODO or link to a bug describing the issue(s) currently preventing type annotation adoption in the BUILD file or in the code itself as appropriate.</p><p> </p><p></p><h2><a class="h" name="3-Python-Style-Rules" href="#3-Python-Style-Rules"><span></span></a><a class="h" name="3-python-style-rules" href="#3-python-style-rules"><span></span></a>3 Python Style Rules</h2><p> </p><p></p><h3><a class="h" name="3_1-Semicolons" href="#3_1-Semicolons"><span></span></a><a class="h" name="3_1-semicolons" href="#3_1-semicolons"><span></span></a>3.1 Semicolons</h3><p>Do not terminate your lines with semicolons, and do not use semicolons to put two statements on the same line.</p><p> </p><p></p><h3><a class="h" name="3_2-Line-length" href="#3_2-Line-length"><span></span></a><a class="h" name="3_2-line-length" href="#3_2-line-length"><span></span></a>3.2 Line length</h3><p>Maximum line length is <em>80 characters</em>.</p><p>Explicit exceptions to the 80 character limit:</p><ul><li>Long import statements.</li><li>URLs, pathnames, or long flags in comments.</li><li>Long string module level constants not containing whitespace that would be inconvenient to split across lines such as URLs or pathnames.</li><li>Pylint disable comments. (e.g.: <code class="code"># pylint: disable=invalid-name</code>)</li></ul><p>Do not use backslash line continuation except for <code class="code">with</code> statements requiring three or more context managers.</p><p>Make use of Python&#39;s <a href="http://docs.python.org/reference/lexical_analysis.html#implicit-line-joining">implicit line joining inside parentheses, brackets and braces</a>. A docstring is a string that is the first statement in a package, module, class or function. These strings can be extracted automatically through the <code class="code">__doc__</code> member of the object and are used by <code class="code">pydoc</code>. (Try running <code class="code">pydoc</code> on your module to see how it looks.) Always use the three double-quote <code class="code">&quot;&quot;&quot;</code> format for docstrings (per <a href="https://www.google.com/url?sa=D&amp;q=http://www.python.org/dev/peps/pep-0257/">PEP 257</a>). A docstring should be organized as a summary line (one physical line) terminated by a period, question mark, or exclamation point, followed by a blank line, followed by the rest of the docstring starting at the same cursor position as the first quote of the first line. There are more formatting guidelines for docstrings below.</p><p>  </p><p></p><h4><a class="h" name="3_8_2-Modules" href="#3_8_2-Modules"><span></span></a><a class="h" name="3_8_2-modules" href="#3_8_2-modules"><span></span></a>3.8.2 Modules</h4><p>Every file should contain license boilerplate. Choose the appropriate boilerplate for the license used by the project (for example, Apache 2.0, BSD, LGPL, GPL)</p><p>Files should start with a docstring describing the contents and usage of the module.</p><pre class="code"><span class="str">&quot;&quot;&quot;A one line summary of the module or program, terminated by a period.

Leave one blank line.  The rest of this docstring should contain an
overall description of the module or program.  Optionally, it may also
contain a brief description of exported classes and functions and/or usage
examples.

  Typical usage example:

  foo = ClassFoo()
  bar = foo.FunctionBar()
&quot;&quot;&quot;</span><span class="pln">
</span></pre><p>  </p><p></p><h4><a class="h" name="3_8_3-Functions-and-Methods" href="#3_8_3-Functions-and-Methods"><span></span></a><a class="h" name="3_8_3-functions-and-methods" href="#3_8_3-functions-and-methods"><span></span></a>3.8.3 Functions and Methods</h4><p>In this section, &ldquo;function&rdquo; means a method, function, or generator.</p><p>A function must have a docstring, unless it meets all of the following criteria:</p><ul><li>not externally visible</li><li>very short</li><li>obvious</li></ul><p>A docstring should give enough information to write a call to the function without reading the function&lsquo;s code. The docstring should be descriptive-style (<code class="code">&quot;&quot;&quot;Fetches rows from a Bigtable.&quot;&quot;&quot;</code>) rather than imperative-style (<code class="code">&quot;&quot;&quot;Fetch rows from a Bigtable.&quot;&quot;&quot;</code>), except for <code class="code">@property</code> data descriptors, which should use the same style as attributes. A docstring should describe the function&rsquo;s calling syntax and its semantics, not its implementation. For tricky code, comments alongside the code are more appropriate than using docstrings.</p><p>A method that overrides a method from a base class may have a simple docstring sending the reader to its overridden method&lsquo;s docstring, such as <code class="code">&quot;&quot;&quot;See base class.&quot;&quot;&quot;</code>. The rationale is that there is no need to repeat in many places documentation that is already present in the base method&rsquo;s docstring. However, if the overriding method&#39;s behavior is substantially different from the overridden method, or details need to be provided (e.g., documenting additional side effects), a docstring with at least those differences is required on the overriding method.</p><p>Certain aspects of a function should be documented in special sections, listed below. Each section begins with a heading line, which ends with a colon. All sections other than the heading should maintain a hanging indent of two or four spaces (be consistent within a file). These sections can be omitted in cases where the function&#39;s name and signature are informative enough that it can be aptly described using a one-line docstring.</p><p> <a href="#doc-function-args"><em>Args:</em></a> :   List each parameter by name. A description should follow the name, and be separated by a colon and a space. length argument lists) and/or `**bar` (arbitrary keyword arguments), they
should be listed as `*foo` and `**bar`.

You may use &ldquo;e&rdquo; as an exception identifier in try/except statements.</li><li>dashes (<code class="code">-</code>) in any package/module name</li><li><code class="code">__double_leading_and_trailing_underscore__</code> names (reserved by Python)</li></ul><p> </p><p></p><h4><a class="h" name="3_16_2-Naming-Conventions" href="#3_16_2-Naming-Conventions"><span></span></a><a class="h" name="3_16_2-naming-conventions" href="#3_16_2-naming-conventions"><span></span></a>3.16.2 Naming Conventions</h4><ul><li><p>&ldquo;Internal&rdquo; means internal to a module, or protected or private within a class.</p></li><li><p>Prepending a single underscore (<code class="code">_</code>) has some support for protecting module variables and functions (not included with <code class="code">from module import *</code>). Unlike Java, there is no need to limit yourself to one class per module.</p></li><li><p>Use CapWords for class names, but lower_with_under.py for module names. Although there are some old modules named CapWords.py, this is now discouraged because it&#39;s confusing when the module happens to be named after a class. (&ldquo;wait -- did I write <code class="code">import StringIO</code> or <code class="code">from StringIO import StringIO</code>?&rdquo;)</p></li><li><p>Underscores may appear in <em>unittest</em> method names starting with <code class="code">test</code> to separate logical components of the name, even if those components use CapWords. One possible pattern is <code class="code">test&lt;MethodUnderTest&gt;_&lt;state&gt;</code>; for example <code class="code">testPop_EmptyStack</code> is okay. There is no One Correct Way to name test methods.</p></li></ul><p> </p><p></p><h4><a class="h" name="3_16_3-File-Naming" href="#3_16_3-File-Naming"><span></span></a><a class="h" name="3_16_3-file-naming" href="#3_16_3-file-naming"><span></span></a>3.16.3 File Naming</h4><p>Python filenames must have a <code class="code">.py</code> extension and must not contain dashes (<code class="code">-</code>). This allows them to be imported and unittested. If you want an executable to be accessible without the extension, use a symbolic link or a simple bash wrapper containing <code class="code">exec &quot;$0.py&quot; &quot;$@&quot;</code>.</p><p> </p><p></p><h4><a class="h" name="3_16_4-Guidelines-derived-from-Guido_s-Recommendations" href="#3_16_4-Guidelines-derived-from-Guido_s-Recommendations"><span></span></a><a class="h" name="3_16_4-guidelines-derived-from-guido_s-recommendations" href="#3_16_4-guidelines-derived-from-guido_s-recommendations"><span></span></a>3.16.4 Guidelines derived from Guido&#39;s Recommendations</h4><p>While Python supports making things private by using a leading double underscore <code class="code">__</code> (aka. &ldquo;dunder&rdquo;) prefix on a name, this is discouraged. Prefer the use of a single underscore. They are easier to type, read, and to access from small unittests. Lint warnings take care of invalid access to protected members.</p><p> </p><p></p><h3><a class="h" name="3_17-Main" href="#3_17-Main"><span></span></a><a class="h" name="3_17-main" href="#3_17-main"><span></span></a>3.17 Main</h3><p>Even a file meant to be used as an executable should be importable and a mere import should not have the side effect of executing the program&#39;s main functionality. The main functionality should be in a <code class="code">main()</code> function.</p><p>In Python, <code class="code">pydoc</code> as well as unit tests require modules to be importable. Your code should always check <code class="code">if __name__ == &#39;__main__&#39;</code> before executing your main program so that the main program is not executed when the module is imported.</p><pre class="code"><span class="kwd">def</span><span class="pln"> main</span><span class="pun">():</span><span class="pln">
    </span><span class="pun">...</span><span class="pln">

    main</span><span class="pun">()</span><span class="pln">
</span></pre><p>All code at the top level will be executed when the module is imported. If a function exceeds about 40 lines, think about whether it can be broken up without harming the structure of the program.</p><p>Even if your long function works perfectly now, someone modifying it in a few months may add new behavior. This could result in bugs that are hard to find. Keeping your functions short and simple makes it easier for other people to read and modify your code.</p><p>You could find long and complicated functions when working with some code. Do not be intimidated by modifying existing code: if working with such a function proves to be difficult, you find that errors are hard to debug, or you want to use a piece of it in several different contexts, consider breaking up the function into smaller and more manageable pieces.</p><p> </p><p></p><h3><a class="h" name="3_19-Type-Annotations" href="#3_19-Type-Annotations"><span></span></a><a class="h" name="3_19-type-annotations" href="#3_19-type-annotations"><span></span></a>3.19 Type Annotations</h3><p> </p><p></p><h4><a class="h" name="3_19_1-General-Rules" href="#3_19_1-General-Rules"><span></span></a><a class="h" name="3_19_1-general-rules" href="#3_19_1-general-rules"><span></span></a>3.19.1 General Rules</h4><ul><li>Familiarize yourself with <a href="https://www.python.org/dev/peps/pep-0484/">PEP-484</a>.</li><li>In methods, only annotate <code class="code">self</code>, or <code class="code">cls</code> if it is necessary for proper type information. e.g., <code class="code">@classmethod def create(cls: Type[T]) -&gt; T: return cls()</code></li><li>If any other variable or a returned type should not be expressed, use <code class="code">Any</code>.</li><li>You are not required to annotate all the functions in a module.<ul><li>At least annotate your public APIs.</li><li>Use judgment to get to a good balance between safety and clarity on the one hand, and flexibility on the other.</li><li>Annotate code that is prone to type-related errors (previous bugs or complexity).</li><li>Annotate code that is hard to understand.</li><li>Annotate code as it becomes stable from a types perspective. </pre><p> </p><p></p><h4><a class="h" name="3_19_9-Tuples-vs-Lists" href="#3_19_9-Tuples-vs-Lists"><span></span></a><a class="h" name="3_19_9-tuples-vs-lists" href="#3_19_9-tuples-vs-lists"><span></span></a>3.19.9 Tuples vs Lists</h4><p>Unlike Lists, which can only have a single type, Tuples can have either a single repeated type or a set number of elements with different types. The latter is commonly used as return type from a function.</p><pre class="code"><span class="pln">a </span><span class="pun">=</span><span class="pln"> </span><span class="pun">[</span><span class="lit">1</span><span class="pun">,</span><span class="pln"> </span><span class="lit">2</span><span class="pun">,</span><span class="pln"> </span><span class="lit">3</span><span class="pun">]</span><span class="pln">  </span><span class="com"># type: List[int]</span><span class="pln">
b </span><span class="pun">=</span><span class="pln"> </span><span class="pun">(</span><span class="lit">1</span><span class="pun">,</span><span class="pln"> </span><span class="lit">2</span><span class="pun">,</span><span class="pln"> </span><span class="lit">3</span><span class="pun">)</span><span class="pln">  </span><span class="com"># type: Tuple[int, ...]</span><span class="pln">
c </span><span class="pun">=</span><span class="pln"> </span><span class="pun">(</span><span class="lit">1</span><span class="pun">,</span><span class="pln"> </span><span class="str">&quot;2&quot;</span><span class="pun">,</span><span class="pln"> </span><span class="lit">3.5</span><span class="pun">)</span><span class="pln">  </span><span class="com"># type: Tuple[int, Text, float]</span><span class="pln">
</span></pre><p>  </p><p></p><h4><a class="h" name="3_19_10-TypeVars" href="#3_19_10-TypeVars"><span></span></a><a class="h" name="3_19_10-typevars" href="#3_19_10-typevars"><span></span></a>3.19.10 TypeVars</h4><p>The Python type system has <a href="https://www.python.org/dev/peps/pep-0484/#generics">generics</a>. The factory function <code class="code">TypeVar</code> is a common way to use them.</p><p>Example:</p><pre class="code"><span class="kwd">from</span><span class="pln"> typing </span><span class="kwd">import</span><span class="pln"> </span><span class="typ">List</span><span class="pun">,</span><span class="pln"> </span><span class="typ">TypeVar</span><span class="pln">
T </span><span class="pun">=</span><span class="pln"> </span><span class="typ">TypeVar</span><span class="pun">(</span><span class="str">&quot;T&quot;</span><span class="pun">)</span><span class="pln">
</span><span class="pun">...</span><span class="pln">
</span><span class="kwd">def</span><span class="pln"> next</span><span class="pun">(</span><span class="pln">l</span><span class="pun">:</span><span class="pln"> </span><span class="typ">List</span><span class="pun">[</span><span class="pln">T</span><span class="pun">])</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> T</span><span class="pun">:</span><span class="pln">
  </span><span class="kwd">return</span><span class="pln"> l</span><span class="pun">.</span><span class="pln">pop</span><span class="pun">()</span><span class="pln">
</span></pre><p>A TypeVar can be constrained:</p><pre class="code"><span class="typ">AddableType</span><span class="pln"> </span><span class="pun">=</span><span class="pln"> </span><span class="typ">TypeVar</span><span class="pun">(</span><span class="str">&quot;AddableType&quot;</span><span class="pun">,</span><span class="pln"> int</span><span class="pun">,</span><span class="pln"> float</span><span class="pun">,</span><span class="pln"> </span><span class="typ">Text</span><span class="pun">)</span><span class="pln">
</span><span class="kwd">def</span><span class="pln"> add</span><span class="pun">(</span><span class="pln">a</span><span class="pun">:</span><span class="pln"> </span><span class="typ">AddableType</span><span class="pun">,</span><span class="pln"> b</span><span class="pun">:</span><span class="pln"> </span><span class="typ">AddableType</span><span class="pun">)</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> </span><span class="typ">AddableType</span><span class="pun">:</span><span class="pln">
  </span><span class="kwd">return</span><span class="pln"> a </span><span class="pun">+</span><span class="pln"> b
</span></pre><p>A common predefined type variable in the <code class="code">typing</code> module is <code class="code">AnyStr</code>. Use it for multiple annotations that can be <code class="code">bytes</code> or <code class="code">unicode</code> and must all be the same type.</p><pre class="code"><span class="kwd">from</span><span class="pln"> typing </span><span class="kwd">import</span><span class="pln"> </span><span class="typ">AnyStr</span><span class="pln">
</span><span class="kwd">def</span><span class="pln"> check_length</span><span class="pun">(</span><span class="pln">x</span><span class="pun">:</span><span class="pln"> </span><span class="typ">AnyStr</span><span class="pun">)</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> </span><span class="typ">AnyStr</span><span class="pun">:</span><span class="pln">
  </span><span class="kwd">if</span><span class="pln"> len</span><span class="pun">(</span><span class="pln">x</span><span class="pun">)</span><span class="pln"> </span><span class="pun">&lt;=</span><span class="pln"> </span><span class="lit">42</span><span class="pun">:</span><span class="pln">
    </span><span class="kwd">return</span><span class="pln"> x
  </span><span class="kwd">raise</span><span class="pln"> </span><span class="typ">ValueError</span><span class="pun">()</span><span class="pln">
</span></pre><p> </p><p></p><h4><a class="h" name="3_19_11-String-types" href="#3_19_11-String-types"><span></span></a><a class="h" name="3_19_11-string-types" href="#3_19_11-string-types"><span></span></a>3.19.11 String types</h4><p>The proper type for annotating strings depends on what versions of Python the code is intended for.</p><p>For Python 3 only code, prefer to use <code class="code">str</code>. <code class="code">Text</code> is also acceptable. Be consistent in using one or the other.</p><p>For Python 2 compatible code, use <code class="code">Text</code>. In some rare cases, <code class="code">str</code> may make sense; typically to aid compatibility when the return types aren&lsquo;t the same between the two Python versions. Avoid using <code class="code">unicode</code>: it doesn&rsquo;t exist in Python 3.</p><p>The reason this discrepancy exists is because <code class="code">str</code> means different things depending on the Python version.</p><pre class="code"><span class="typ">No</span><span class="pun">:</span><span class="pln">
</span><span class="kwd">def</span><span class="pln"> py2_code</span><span class="pun">(</span><span class="pln">x</span><span class="pun">:</span><span class="pln"> str</span><span class="pun">)</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> unicode</span><span class="pun">:</span><span class="pln">
  </span><span class="pun">...</span><span class="pln">
</span></pre><p>For code that deals with binary data, use <code class="code">bytes</code>.</p><pre class="code"><span class="kwd">def</span><span class="pln"> deals_with_binary_data</span><span class="pun">(</span><span class="pln">x</span><span class="pun">:</span><span class="pln"> bytes</span><span class="pun">)</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> bytes</span><span class="pun">:</span><span class="pln">
  </span><span class="pun">...</span><span class="pln">
</span></pre><p>For Python 2 compatible code that processes text data (<code class="code">str</code> or <code class="code">unicode</code> in Python 2, <code class="code">str</code> in Python 3), use <code class="code">Text</code>. For Python 3 only code that process text data, prefer <code class="code">str</code>.</p><pre class="code"><span class="kwd">from</span><span class="pln"> typing </span><span class="kwd">import</span><span class="pln"> </span><span class="typ">Text</span><span class="pln">
</span><span class="pun">...</span><span class="pln">
</span><span class="kwd">def</span><span class="pln"> py2_compatible</span><span class="pun">(</span><span class="pln">x</span><span class="pun">:</span><span class="pln"> </span><span class="typ">Text</span><span class="pun">)</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> </span><span class="typ">Text</span><span class="pun">:</span><span class="pln">
  </span><span class="pun">...</span><span class="pln">
</span><span class="kwd">def</span><span class="pln"> py3_only</span><span class="pun">(</span><span class="pln">x</span><span class="pun">:</span><span class="pln"> str</span><span class="pun">)</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> str</span><span class="pun">:</span><span class="pln">
  </span><span class="pun">...</span><span class="pln">
</span></pre><p>If the type can be either bytes or text, use <code class="code">Union</code>, with the appropriate text type.</p><pre class="code"><span class="kwd">from</span><span class="pln"> typing </span><span class="kwd">import</span><span class="pln"> </span><span class="typ">Text</span><span class="pun">,</span><span class="pln"> </span><span class="typ">Union</span><span class="pln">
</span><span class="pun">...</span><span class="pln">
</span><span class="kwd">def</span><span class="pln"> py2_compatible</span><span class="pun">(</span><span class="pln">x</span><span class="pun">:</span><span class="pln"> </span><span class="typ">Union</span><span class="pun">[</span><span class="pln">bytes</span><span class="pun">,</span><span class="pln"> </span><span class="typ">Text</span><span class="pun">])</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> </span><span class="typ">Union</span><span class="pun">[</span><span class="pln">bytes</span><span class="pun">,</span><span class="pln"> </span><span class="typ">Text</span><span class="pun">]:</span><span class="pln">
  </span><span class="pun">...</span><span class="pln">
</span><span class="kwd">def</span><span class="pln"> py3_only</span><span class="pun">(</span><span class="pln">x</span><span class="pun">:</span><span class="pln"> </span><span class="typ">Union</span><span class="pun">[</span><span class="pln">bytes</span><span class="pun">,</span><span class="pln"> str</span><span class="pun">])</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> </span><span class="typ">Union</span><span class="pun">[</span><span class="pln">bytes</span><span class="pun">,</span><span class="pln"> str</span><span class="pun">]:</span><span class="pln">
  </span><span class="pun">...</span><span class="pln">
</span></pre><p>If all the string types of a function are always the same, for example if the return type is the same as the argument type in the code above, use <a href="#typing-type-var">AnyStr</a>.</p><p>Writing it like this will simplify the process of porting the code to Python 3.</p><p> </p><p></p><h4><a class="h" name="3_19_12-Imports-For-Typing" href="#3_19_12-Imports-For-Typing"><span></span></a><a class="h" name="3_19_12-imports-for-typing" href="#3_19_12-imports-for-typing"><span></span></a>3.19.12 Imports For Typing</h4><p>For classes from the <code class="code">typing</code> module, always import the class itself. You are explicitly allowed to import multiple specific classes on one line from the <code class="code">typing</code> module. Ex:</p><pre class="code"><span class="kwd">from</span><span class="pln"> typing </span><span class="kwd">import</span><span class="pln"> </span><span class="typ">Any</span><span class="pun">,</span><span class="pln"> </span><span class="typ">Dict</span><span class="pun">,</span><span class="pln"> </span><span class="typ">Optional</span><span class="pln">
</span></pre><p>Given that this way of importing from <code class="code">typing</code> adds items to the local namespace, any names in <code class="code">typing</code> should be treated similarly to keywords, and not be defined in your Python code, typed or not. If there is a collision between a type and an existing name in a module, import it using <code class="code">import x as y</code>.</p><pre class="code"><span class="kwd">from</span><span class="pln"> typing </span><span class="kwd">import</span><span class="pln"> </span><span class="typ">Any</span><span class="pln"> </span><span class="kwd">as</span><span class="pln"> </span><span class="typ">AnyType</span><span class="pln">
</span></pre><p> </p><p></p><h4><a class="h" name="3_19_13-Conditional-Imports" href="#3_19_13-Conditional-Imports"><span></span></a><a class="h" name="3_19_13-conditional-imports" href="#3_19_13-conditional-imports"><span></span></a>3.19.13 Conditional Imports</h4><p>Use conditional imports only in exceptional cases where the additional imports needed for type checking must be avoided at runtime. This pattern is discouraged; alternatives such as refactoring the code to allow top level imports should be preferred.</p><p>Imports that are needed only for type annotations can be placed within an <code class="code">if TYPE_CHECKING:</code> block.</p><ul><li>Conditionally imported types need to be referenced as strings, to be forward compatible with Python 3.6 where the annotation expressions are actually evaluated.</li><li>Only entities that are used solely for typing should be defined here; this includes aliases. Otherwise it will be a runtime error, as the module will not be imported at runtime.</li><li>The block should be right after all the normal imports.</li><li>There should be no empty lines in the typing imports list.</li><li>Sort this list as if it were a regular imports list.</li></ul><pre class="code"><span class="kwd">import</span><span class="pln"> typing
</span><span class="kwd">if</span><span class="pln"> typing</span><span class="pun">.</span><span class="pln">TYPE_CHECKING</span><span class="pun">:</span><span class="pln">
  </span><span class="kwd">import</span><span class="pln"> sketch
</span><span class="kwd">def</span><span class="pln"> f</span><span class="pun">(</span><span class="pln">x</span><span class="pun">:</span><span class="pln"> </span><span class="str">&quot;sketch.Sketch&quot;</span><span class="pun">):</span><span class="pln"> </span><span class="pun">...</span><span class="pln">
</span></pre><p> </p><p></p><h4><a class="h" name="3_19_14-Circular-Dependencies" href="#3_19_14-Circular-Dependencies"><span></span></a><a class="h" name="3_19_14-circular-dependencies" href="#3_19_14-circular-dependencies"><span></span></a>3.19.14 Circular Dependencies</h4><p>Circular dependencies that are caused by typing are code smells. Such code is a good candidate for refactoring. Although technically it is possible to keep circular dependencies, the <a href="#typing-build-deps">build system</a> will not let you do so because each module has to depend on the other.</p><p>Replace modules that create circular dependency imports with <code class="code">Any</code>. Set an <a href="#typing-aliases">alias</a> with a meaningful name, and use the real type name from this module (any attribute of Any is Any). Alias definitions should be separated from the last import by one line.</p><pre class="code"><span class="kwd">from</span><span class="pln"> typing </span><span class="kwd">import</span><span class="pln"> </span><span class="typ">Any</span><span class="pln">

some_mod </span><span class="pun">=</span><span class="pln"> </span><span class="typ">Any</span><span class="pln">  </span><span class="com"># some_mod.py imports this module.</span><span class="pln">
</span><span class="pun">...</span><span class="pln">

</span><span class="kwd">def</span><span class="pln"> my_method</span><span class="pun">(</span><span class="pln">self</span><span class="pun">,</span><span class="pln"> var</span><span class="pun">:</span><span class="pln"> some_mod</span><span class="pun">.</span><span class="typ">SomeType</span><span class="pun">)</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> </span><span class="kwd">None</span><span class="pun">:</span><span class="pln">
  </span><span class="pun">...</span><span class="pln">
</span></pre><p>  </p><p></p><h4><a class="h" name="3_19_15-Generics" href="#3_19_15-Generics"><span></span></a><a class="h" name="3_19_15-generics" href="#3_19_15-generics"><span></span></a>3.19.15 Generics</h4><p>When annotating, prefer to specify type parameters for generic types; otherwise, <a href="https://www.python.org/dev/peps/pep-0484/#the-any-type">the generics&#39; parameters will be assumed to be <code class="code">Any</code></a>.</p><pre class="code"><span class="kwd">def</span><span class="pln"> get_names</span><span class="pun">(</span><span class="pln">employee_ids</span><span class="pun">:</span><span class="pln"> </span><span class="typ">List</span><span class="pun">[</span><span class="pln">int</span><span class="pun">])</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> </span><span class="typ">Dict</span><span class="pun">[</span><span class="pln">int</span><span class="pun">,</span><span class="pln"> </span><span class="typ">Any</span><span class="pun">]:</span><span class="pln">
  </span><span class="pun">...</span><span class="pln">
</span></pre><pre class="code"><span class="com"># These are both interpreted as get_names(employee_ids: List[Any]) -&gt; Dict[Any, Any]</span><span class="pln">
</span><span class="kwd">def</span><span class="pln"> get_names</span><span class="pun">(</span><span class="pln">employee_ids</span><span class="pun">:</span><span class="pln"> list</span><span class="pun">)</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> </span><span class="typ">Dict</span><span class="pun">:</span><span class="pln">
  </span><span class="pun">...</span><span class="pln">

</span><span class="kwd">def</span><span class="pln"> get_names</span><span class="pun">(</span><span class="pln">employee_ids</span><span class="pun">:</span><span class="pln"> </span><span class="typ">List</span><span class="pun">)</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> </span><span class="typ">Dict</span><span class="pun">:</span><span class="pln">
  </span><span class="pun">...</span><span class="pln">
</span></pre><p>If the best type parameter for a generic is <code class="code">Any</code>, make it explicit, but remember that in many cases <a href="#typing-type-var"><code class="code">TypeVar</code></a> might be more appropriate:</p><pre class="code"><span class="kwd">def</span><span class="pln"> get_names</span><span class="pun">(</span><span class="pln">employee_ids</span><span class="pun">:</span><span class="pln"> </span><span class="typ">List</span><span class="pun">[</span><span class="typ">Any</span><span class="pun">])</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> </span><span class="typ">Dict</span><span class="pun">[</span><span class="typ">Any</span><span class="pun">,</span><span class="pln"> </span><span class="typ">Text</span><span class="pun">]:</span><span class="pln">
  </span><span class="str">&quot;&quot;&quot;Returns a mapping from employee ID to employee name for given IDs.&quot;&quot;&quot;</span><span class="pln">
</span></pre><pre class="code"><span class="pln">T </span><span class="pun">=</span><span class="pln"> </span><span class="typ">TypeVar</span><span class="pun">(</span><span class="str">&#39;T&#39;</span><span class="pun">)</span><span class="pln">
</span><span class="kwd">def</span><span class="pln"> get_names</span><span class="pun">(</span><span class="pln">employee_ids</span><span class="pun">:</span><span class="pln"> </span><span class="typ">List</span><span class="pun">[</span><span class="pln">T</span><span class="pun">])</span><span class="pln"> </span><span class="pun">-&gt;</span><span class="pln"> </span><span class="typ">Dict</span><span class="pun">[</span><span class="pln">T</span><span class="pun">,</span><span class="pln"> </span><span class="typ">Text</span><span class="pun">]:</span><span class="pln">
  </span><span class="str">&quot;&quot;&quot;Returns a mapping from employee ID to employee name for given IDs.&quot;&quot;&quot;</span><span class="pln">
</span></pre><p></p><p></p><h2><a class="h" name="4-Parting-Words" href="#4-Parting-Words"><span></span></a><a class="h" name="4-parting-words" href="#4-parting-words"><span></span></a>4 Parting Words</h2><p><em>BE CONSISTENT</em>.</p><p>If you&#39;re editing code, take a few minutes to look at the code around you and determine its style. If they use spaces around all their arithmetic operators, you should too. If their comments have little boxes of hash marks around them, make your comments have little boxes of hash marks around them too.</p><p>The point of having style guidelines is to have a common vocabulary of coding so people can concentrate on what you&lsquo;re saying rather than on how you&rsquo;re saying it. We present global style rules here so people know the vocabulary, but local style is also important. If code you add to a file looks drastically different from the existing code around it, it throws readers out of their rhythm when they go to read it.
